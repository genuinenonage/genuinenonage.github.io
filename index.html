css浮动

CSS的Float（浮动）使元素脱离文档流，按照指定的方向移动（左或右发生移动），直到它的外边缘碰到包含框或另一个浮动框的边框为止。

浮动的样式名：float，值：left向左浮动，right向右浮动，none默认值元素不浮动。

CSS定位

指定元素的定位类型position。

静态定位：static，默认值。

绝对定位：absolute，元素在页面上的固定位置。当前元素会让出原来的位置，其他元素可以占用。

相对定位：relative，相对于自己原来的位置进行定位。

固定定位：fixed，在浏览器窗口固定位置，不会随和页面的上下移动而移动。

CSS盒子模型

所有HTML元素可以看作盒子。可以设置HTML元素周围的内容。

边距（margin），边框（border），填充（padding），和实际内容（content）。

第三章JavaScript

曾经交JavaScript，后来被ECMA权威工会倡导所有浏览器厂商兼容，改名为ECMAScript，简称ES。

Vue3必须需ES6来支持。

JS特点

脚本语言：JS是一种解释型的脚本语言。不同于Java等语言先编译后执行，JS不会产生编译的字节码文件，而是直接在程序运行过程中对源文件进行解释。

弱类型：声明一个变量可以接受任何类型的对象。并且会在程序执行过程中根据上下文自动转换类型。

跨平台性：JS不依赖操作系统，仅需要浏览器支持。

什么是BOM编程

BOM是Browser Object Model的简写，即浏览器对象模型。有一系列对象组成，可以访问、控制、修改浏览器的属性和方法。

BOM编程的对象结构如下

  • window 顶级对象,代表整个浏览器窗口
    • location对象 window对象的属性之一,代表浏览器的地址栏
    • history对象 window对象的属性之一,代表浏览器的访问历史
    • screen对象 window对象的属性之一,代表屏幕
    • navigator对象 window对象的属性之一,代表浏览器软件本身
    • document对象 window对象的属性之一,代表浏览器窗口目前解析的html文档

什么是DOM编程

使用document对象的API完成对网页HTML文档进行动态修改,以实现网页数据和样式动态变化效果的编程。

会话级存储和持久级存储

通过BOM编程实现

会话级存储：浏览器在内存上临时存储的数据，浏览器关闭后，数据失去，通过window的sessionStorge属性实现。

持久级存储：浏览器在磁盘上持久存储的数据，浏览器关闭后，数据仍在，通过window的localStorge实现。

第四章 web后端服务

XML

是EXtensible Markup Language的缩写，可扩展标记语言。XML和HTML一样都是标记语言，它们的基本语法都是标签。

可扩展：XML允许自定义格式。但这不代表可以随便写。

可以使用DOM4J进行XML文件解析。

Tomcat

当前应用最广的JavaWeb服务器。Web服务器通常由硬件和软件共同构成。

Web项目的标准结构

  • static非必要目录，一般在此处放静态资源（CSS JS IMG）。
  • WEB-INF 必要目录，受保护的资源目录，浏览器通过url不可以直接访问。
    • classes 必要目录，src下源代码、配置文件，编译后会在该目录下。
    • lib 必要目录，项目以来的jar编译后会在该目录。
    • web.xml 必要文件，web项目的基本配置文件。
  • index.html 非必要文件，index.html/index.htm/index.jsp为默认的欢迎页

Servlet

Servlet (server applet) 是运行在服务端(tomcat)的Java小程序，是sun公司提供一套定义动态资源规范; 从代码层面上来讲Servlet就是一个接口。

Servlet生命周期

（可以编码的阶段）

  1. 构造器对象
  2. 初始化
  3. 服务处理：仅有多次执行
  4. 销毁

Servlet继承结构

Servlet接口：所有Servlet必须实现接口中的定义的方法。

GenericServlect抽象类：对Servlet接口一些固定功能的粗糙实现,以及对service方法的再次抽象声明,并定义了一些其他相关功能方法。

HttpServlet抽象类：除了基本的实现以外,增加了更多的基础功能。

我们可以自定义实现Servlet的所有方法，也可使用抽象类，专注于业务的开发。

ServletConfig和ServletContext

ServletContext服务共享是单例的，ServletConfig是多例的，每个Servlet单独一个。

ServletConfig为Servlet提供初始配置参数的一种对象。

ServletContext可以为所有的Servlet提供初始配置参数。

请求转发和重定向

Tomcat会将请求报文转换为ServletRequest对象，会将ServletResponse对象转换相应报文。

转发操作的是ServletRequest
重定向操作的是ServlectReponse

乱码问题

根本原因：数据的编码和解码使用的不是同一个字符集，或者使用了不支持某个语言文字的字符集。

MVC（Model View Controller）是软件工程中的一种软件架构模式，它把软件系统分为模型、视图和控制器三个基本部分。用一种业务逻辑、数据、界面显示分离的方法组织代码。

Vite+Vue3项目的目录结构

什么是.vue文件？

传统的页面有.htm .css .js三个文件组成（多文件组件）

vue将这文件合并成一个.vue文件（Single-File Component，SFC，单文件组件），每个.vue文件可以理解为一个组件，多个.vue文件可以构成一个整体页面。

template标签 代表html部分代码

script标签 代表js代码

style标签 代表css代码

工程化vue项目如何组织这些组件？

  • index.html是项目的入口,其中 <div id ='app'></div>是用于挂载所有组建的元素
  • index.html中的script标签引入了一个main.js文件,具体的挂载过程在main.js中执行
  • main.js是vue工程中非常重要的文件,他决定这项目使用哪些依赖,导入的第一个组件
  • App.vue是vue中的核心组件,所有的其他组件都要通过该组件进行导入,该组件通过路由可以控制页面的切换

响应式基础

关键字ref

ref 可以将一个基本类型的数据（如字符串，数字等）转换为一个响应式对象。 ref 只能包裹单一元素。

关键字reactive

使用 reactive() 函数创建一个响应式对象或数组。

条件渲染

  • v-if='表达式' 只会在指令的表达式返回真值时才被渲染
  • 一个 v-else 元素必须跟在一个 v-if 元素后面，否则它将不会被识别。

v-show也是条件渲染，v-show 仅切换了该元素上名为 display 的 CSS 属性。

双向绑定

属性计算

computed() 方法期望接收一个 getter 函数，返回值为一个计算属性 ref。普通函数多次调用会执行多次，属性计算，属性不变时，多次调用只会调用一次。

数据监听器

使用 watch 函数在每次响应式状态发生变化时触发回调函数：

  • 当数据发生变化时需要执行相应的操作
  • 监听数据变化，当满足一定条件时触发相应操作
  • 在异步操作前或操作后需要执行相应的操作

watchEffect默认监听所有的响应式数据

Vue生命周期

常见钩子函数

  • onMounted() 注册一个回调函数，在组件挂载完成后执行。
  • onUpdated() 注册一个回调函数，在组件因为响应式状态变更而更新其 DOM 树之后调用。
  • onUnmounted() 注册一个回调函数，在组件实例被卸载之后调用。
  • onBeforeMount() 注册一个钩子，在组件被挂载之前被调用。
  • onBeforeUpdate() 注册一个钩子，在组件即将因为响应式状态变更而更新其 DOM 树之前调用。
  • onBeforeUnmount() 注册一个钩子，在组件实例被卸载之前调用。

第一章高性能架构模式

海量用户和海量数据的特点，单个数据库服务器难以满足业务需要，必须考虑数据库集群的方式来提升性能。

高性能数据库集群的第一种方式—-读写分离，第二种方式—-数据库分片。

1. 读写分离架构

基本原理使将数据库读写操作分散到不同的节点上。

基本实现：主库负责事务性的增删改操作，从库负责查询操作。将读写操作路由到不同的数据库。

能够有效的避免有数据更新导致的行锁，整个系统的查询性能极大提升。

CAP定理：在一个分布式系统中，设计读写操作时。只能保证一致性、可用性、分区容错性三者中的两个。

C一致性：对于客户端来说，读操作能保证返回最新的写操作结果。

A可用性：非故障节点在合理的时间返回合理的响应。

P分区容错性：出现网络分区（网络丢包延迟等）后，系统能够继续履行职责。

2. 数据库分片结构

读写分离分散了数据库读写操作的压力，没有分散存储压力。这就需要将存储分散到多台数据库服务器上。

数据分片：将单一数据库的数据，分散存放到多个数据库或表中。数据分片的有效手段是对关系型数据库进行分库分表。

数据分片又分为垂直分片和水平分片。

2.1. 垂直分片

垂直分库：按业务拆分。专库专用。按照业务将表归类，分布到不同的数据库中，将压力分散到不同的数据库。

可以缓解数据量和访问量带来的问题，但无法根治。表中数据依旧会超过单节点阈值。

垂直分表：将表中不常用的列，或者占用大量空间的列拆分出去，用单独表存储。适合数据行数不多，存储特别大的表。

水平分表适合表行数特别大的表，水平分表属于水平分片。

2.2. 水平分片

通过某个字段或某几个字段，根据某种规则将数据分散到多个库或表中。

水平分表：单表拆分多表后，在同一个数据库服务器中。如果性能能够满足业务要求，可以不拆分到多台数据库服务器。

水平分库：单表拆分多表后，单台服务器依旧无法满足性能要求，需要多表分散到不同的数据库服务器中。

3. 读写分离和数据分片架构

4. 实现方式

4.1 程序代码封装，通过业务代码，抽象一个数据访问层，实现读写分离和数据库服务连接的管理。

4.2 中间件封装，独立一套系统出来，实现读写分离和数据库服务器连接的管理。对于业务服务器，访问中间件和访问数据库没有区别。

埋点与日志的区别是什么？

发表于 2023-04-27

  1. 代码里有专门打日志的代码，也有专门打埋点的。

  2. 主要作用：日志用于排查问题，埋点用于记录用户行为数据。

埋点

埋点主要是了解用户行为的重要步骤，注入特定的代码以记录关键指标用于未来提升应用性能的重要环节。

埋点与日志的区别

  1. 埋点可以看作是日志的子集。被埋点的任何数据都应该记录在日志中。

  2. 埋点承担了为聚合分析发布关键性能数据的职责。

  3. 日志则提供了用户在不同级别跟踪应用的细节信息。

    从低到高依次为:

    Verobse:几乎提供了所有的细节，主要用于跟踪执行过程中控制流

    Debug:表示数据主要用于调试

    Info:表示非错误信息

    Warning:表示可恢复的错误

    Error：表示不可恢复的错误

  4. 日志的记录会贯穿应用的整个生命周期，而埋点只应该用在某些特定阶段。

  5. 埋点数据会发送到服务器，日志是记录在本地设备。

  6. 埋点与日志不可相互替代

  7. 日志可以非常详细。埋点应该尽可能少地，因为向服务器报告时会消耗网络资源。

因此，只对有需要的事件进行埋点非常重要，这些事件要包含足够多的数据以满足重要报告的需要，埋点和过度埋点之间并没有清晰的界限。一开始应仅对少量报告进行埋点，然后随着时间推移，逐步增加埋点的覆盖率。

埋点上传机制

实时上报：服务端提供日志记录接口，当触发事件时，直接调用日志记录接口将日志记录在服务端。如果是频率低，数据量小，实时性要求高的数据可以不设限制。

延时上报：服务端提供日志上传接口，客户端将日志暂存本地，当达到一定大小或一定时间时，将日志通过上传接口压缩后上传。这个时间和数据量一般根据公司业务情况自定义。

关键数据后端埋点

关键行为采集使用后端埋点。比如订单支付、注册成功与否，通过后端数据采集，好处是：数据量全面和提高数据的准确性、实时性。

完善的埋点规范

比如我们规定埋点名称只能由字母、下划线、数字组成，并保证其的唯一性。例如，设定ck是点击、sw是展示；埋点事件命名规范是：{团队|业务|角色}{组件|页面}{具体元素}_{动作}

示例：

patent_comt_share_ck 。patent代表业务线，comt评价组件，share分享按钮，ck点击

  1. 用户名密码调用认证中心登录接口

  2. 认证中心服务器操作：

    1. 登录之前，先退出其他已登录账号

      1. 删除登录凭证TGT和缓存的用户数据

      2. 删除凭证Cookie

      3. 删除所有TGT对应的调用凭证，并通知客户端登出注销本地Session

        1. 删除TGT
        2. 获取jwtToken
        3. 把jwtToken存入黑名单
        4. 响应客户端，登出请求
        5. 删除jwtToken

    2. 根据用户名密码查询用户信息。

    3. 获取TGT登录凭证

      1. 根据UUID随机生成TGT
      2. TGT为key，将用户信息缓存Redis
      3. TGT存入Cookie

    4. 生成授权码code，跳转重定向地址

      1. 根据appId获取app地址
      2. 根据UUID生成随机code
      3. code为key，将TGT登录凭证和app地址，缓存Redis
      4. 将code拼接到重定向地址：业务页面地址+code

    5. 接口响应重定向地址

  3. 重定向业务页面

  4. 业务页面调用业务后端服务

  5. 业务后端服务器根据code调用认证中心Oauth2获取token。（认证中心Oauth2同时支持code和用户名密码两种登录获取token方式）

  6. 认证中心获取token接口

    1. 验证参数与应用
    2. 校验授权，通过code，获取包含user等信息的AccessTokenContent对象
      1. code获取缓存TGT
      2. 根据TGT获取缓存用户信息
      3. 从用户中心获取用户角色和权限信息
      4. 分别缓存用户角色和权限信息
    3. 生成jwtToken并返回
      1. 根据TGT、用户信息、appId、业务地址
      2. 根据UUID随机生成accessToken调用凭证
      3. accessToken为key缓存TGT、用户信息、appId、业务地址信息
      4. TGT为key缓存accessToken调用凭证
      5. 根据UUID随机生成resfreshToken刷新调用凭证
      6. resfreshToken为key缓存TGT、用户信息、appId、业务地址、accessToken
      7. 创建token对象（accessToken调用凭证，resfreshToken：当前AccessToken超时用于刷新AccessToken并延长服务端session时效必要参数，应用标识appId，用户信息user）
      8. 将token对象生成JWT
        1. String jwt = Jwts.builder().setClaims(claims).signWith(SignatureAlgorithm.HS256, secret).compact();
        2. 将accessToken为key，缓存jwt
      9. 响应业务系统JWT
      10. 业务系统响应前端token
        1. 业务系统通过response.getOutputStream().write(JSONObject.toJSONString(respResult).getBytes(StandardCharsets.UTF_8))响应前端token
      11. 业务前端根据token调用业务系统接口
      12. 业务系统接口每次都会经过权限拦截，获取接口token，调用认证中心接口/checkToken，判断token是否在黑名单，不在存表示token有效。

http是无状态的吗？怎么解决的？

发表于 2023-04-25 分类于 web

为什么说http是无状态的？它不是一直在维持tcp的长连接吗？

什么是http无状态

http的每次请求都是一次单独的请求。

http无状态的优点

不需要额外资源记录状态信息。实现简单、服务器负担少。

对于有状态的协议，如果将会话状态与连接绑定在一起，意外断开，整个会话就会丢失。重新连接一般需要从头开始。

http无状态的缺点

每次请求都是独立的，所以每个请求需要的所有信息都必须包含在请求中一次发送到服务端。导致消息结构比较复杂，必须能够支持大量元数据。

解决http无状态的方案

Cookie和Session

身份验证流程：用户登录到服务器验证成功后，相关数据保存在当前会话中，服务器发挥session_id，写入到用户的cookie中。后续的请求需要在Cookie中取出session_id传给服务器，服务器根据session_id检索之前保存的数据，确认用户身份。

弊端：Session信息存储在服务器，必然占用内存。用户多了后，开销增大，为了提高效率，需要分布式、负载均衡。

Token

Token传递过程跟Cookie类似，只是传递对象变成了Token。

  1. 用户使用用户名、密码请求服务器后。
  2. 服务器生成 Token并响应给客户端。
  3. 客户端再次请求时带上 Token，服务器取到 Token 进行认证鉴权。
  4. 认证通过，服务器响应数据。认证不通过，跳转登陆。

JWT

JWT时不需要查库的，所需的基础信息可以放在JWT中，服务器只需要判断用户是否可以拥有接口权限，JWT的内容不是无限的，其他信息需要通过id查询数据库。

  1. 客户端使用账户密码请求登录接口
  2. 服务器使用签名密钥生成JWT ,然后返回JWT给客户端
  3. 客户端再次向服务端请求其他接口时带上JWT
  4. 服务端接收到JWT后验证签名的有效性.对客户端做出相应的响应

和Token的区别

token需要查库验证token是否有效，JWT不用查库或少查库，只需在服务器验证。用户的信息已被加密其中，只需要校验即可，校验JWT自己实现。

为什么JWT不用验证token是否有效

JWT查询流程

  1. 解析JWT
  2. 判断签名是否正确，根据生成签名时使用的密钥和加密算法，这步之后说明是可信的。
  3. 判断JWT token是否过期，根据exp，判断是否是VIP，根据isVip

动态代理

发表于 2023-04-24 分类于 Java

首先了解下静态代理。

静态代理

目标类和代理类实现了相同的接口，并且代理类中依赖了目标类。在代理类的方法中调用目标类方法，并做一些增强性的工作。

静态代理的缺陷

需要手动为每个目标类编写对应的代理类。如果当前系统已经有成百上千个类，工作量太大了。

当接口改变时，不仅目标类需要改变，所有的代理类都需要进行相应的变化。

目标：如何少写或者不写代理类，却能完成代理功能。

动态代理

在程序的执行过程中，使用jdk的反射机制，创建代理对象，并动态的指定代理的目标类。

常用的两种实现方式

jdk代理

cjlib代理

jdk动态代理

基于Java的反射机制实现，使用jdk反射包下的Proxy和InvocationHandler实现代理对象的动态创建。（jdk动态代理要求目标对象必须实现接口）

InvocationHandler接口

接口中只有一个方法 ：invoke()，代理类要完成的功能写在invoke()方法中。

代理类需要的工作：

调用目标类的方法。

功能增强，在目标方法调用前后，增加功能。

proxy类

最核心的类，使用静态方法 newProxyInstance() ，创建代理对象。

cjlib动态代理

cglib是第三方的工具库，可以创建代理对象。

cglib的原理是继承，cglib通过继承目标类，创建它的子类，在子类中重写父类中的方法，实现功能的修改。

cglib要求目标类不能是final，且目标类的方法不能是final修饰的。

Cookie和Session

发表于 2023-02-23 更新于 2023-03-09 分类于 Web

会话跟踪是Web程序中常用的技术，用来跟踪用户的整个会话。常用的会话跟踪技术是Cookie和Session。

Cookie

基本概念

Cookie是某些网站为了辨别用户身份，进行Session跟踪而存储在用户本地终端上的数据（通常经过加密），由用户客户端计算机暂时或永久保存的信息。

存储类型

内存式：存储在内存中，浏览器关闭后就会消失，由于其存储时间较短，因此也被称为非持久Cookie或会话Cookie。

硬盘式：保存在硬盘中，不会随浏览器的关闭而消失，除非用户手工清理或到了过期时间。由于硬盘式Cookie存储时间是长期的，因此也被称为持久Cookie。

工作原理

Cookie定义了一些HTTP请求头和HTTP响应头，通过这些HTTP头信息使服务器与客户进行状态交互。

客户端请求服务器后，如果服务器需要记录用户状态，服务器会在响应信息中包含一个Set-Cookie的响应头，客户端会根据这个响应头存储Cookie信息。再次请求服务器时，客户端会在请求信息中包含一个Cookie请求头，而服务器会根据这个请求头进行用户身份、状态等较验。

在服务器响应请求中，响应头中包一个Set-Cookie字段。

Cookie的属性设置

Java中把Cookie封装成了javax.servlet.http.Cookie类。服务器通过操作Cookie类对象对客户端Cookie进行操作。

通过request.getCookie()获取客户端提交的所有Cookie，通过response.addCookie()向客户端设置Cookie。

每个属性对应一个getter方法与一个setter方法。

Unicode编码：保存中文

中文与英文字符不同，中文属于Unicode字符，在内存中占4个字符，而英文属于ASCII字符，内存中只占2个字节。Cookie中使用Unicode字符时需要对Unicode字符进行编码，否则会乱码。

Cookie中保存中文只能编码。一般使用UTF-8编码即可。不推荐使用GBK等中文编码，因为浏览器不一定支持，而且JavaScript也不支持GBK编码

设置Cookie的有效期

从客户端读取Cookie时，包括maxAge在内的其他属性都是不可读的，也不会被提交。浏览器提交Cookie时只会提交name与value属性。maxAge属性只被浏览器用来判断Cookie是否过期。

修改和删除cookie

要想修改Cookie只能使用一个同名的Cookie来覆盖原来的Cookie，达到修改的目的。删除时只需要把maxAge修改为0即可。

修改、删除Cookie时，新建的Cookie除value、maxAge之外的所有属性，例如name、path、domain等，都要与原Cookie完全一样。否则，浏览器将视为两个不同的Cookie不予覆盖，导致修改、删除失败。

设置cookie的域名

Cookie是不可跨域名的。域名 www.google.com 颁发的Cookie不会被提交到域名 www.baidu.com 去。这是由Cookie的隐私安全机制决定的。隐私安全机制能够禁止网站非法获取其他网站的Cookie。

设置cookie的安全属性

secure属性并不能对Cookie内容加密，因而不能保证绝对的安全性。如果需要高安全性，需要在程序中对Cookie内容加密、解密，以防泄密。

Session

客户端浏览器访问服务器的时候，服务器把客户端信息以某种形式记录在服务器上。这就是Session。

工作原理

某个客户端初次请求创建Session时，服务器先检查请求中是否包含Session标识，成为session id，包含说明已经为此客户端创建过session，服务器根据session id把Session检索出来，不包含则需要创建一个唯一session id，并在本次响应中返回给客户端保存。

保存session id的方式

使用Cookie保存

在交互过程中浏览器可以自动的按照规则把这个标识发挥给服务器。一般这个cookie的名字都是类似于SEEESIONID。然而，比如weblogic对于web应用程序生成的cookie，JSESSIONID=ByOK3v，它的名字就是JSESSIONID。

URL重写

由于cookie可以被人为的禁止，必须有其他机制以便在cookie被禁止时仍然能够把session id传递回服务器。经常被使用的一种技术叫做URL重写，就是把session id直接附加在URL路径的后面。一种是作为URL路径的附加信息http://…../xxx;jsessionid=ByOK3v，另一种是作为查询字符串附加在URL后面http://…../xxx?jsessionid=ByOK3v。

表单隐藏字段

表单隐藏就是服务器会自动修改表单，添加一个隐藏字段，以便在表单提交时能够把session id传递回服务器。

Session的常用方法

Session对应的类为javax.servlet.http.HttpSession类。Session对象是在客户端第一次请求服务器的时候创建的。Session也是一种key-value的属性对，通过getAttribute(String key)和setAttribute(String key，Objectvalue)方法读写客户状态信息。Servlet里通过request.getSession()方法获取该客户的Session。

当多个客户端执行程序时，服务器会保存多个客户端的Session。获取Session的时候也不需要声明获取谁的Session。Session机制决定了当前客户只会获取到自己的Session，而不会获取到别人的Session。

Session保存在服务器端。为了获得更高的存取速度，服务器一般把Session放在内存里。每个用户都会有一个独立的Session。如果Session内容过于复杂，当大量客户访问服务器时可能会导致内存溢出。因此，Session里的信息应该尽量精简。

Session和Cookie的区别与联系

区别

  1. cookie数据存放在客户端，session数据放在服务器上。
  2. cookie不是很安全，别人可以分析存放在本地的cookie并进行cookie欺骗，考虑到安全应当使用session。
  3. session会在一定时间内保存在服务器上，当访问增多，会比较占用你服务器的性能，考虑性能应当使用cookie。
  4. 不同浏览器对cookie的数据大小限制不同，个数限制也不相同。单个cookie保存的数据不能超过4K，很多浏览器都限制一个站点最多保存20个cookie。
  5. 可以考虑将登陆信息等重要信息存放为session，不重要的信息可以放在cookie中。

联系

  1. 都是用来记录用户的信息，以便让服务器分辨不同的用户。
  2. 可以搭配使用，但都有自己的使用局限，要考虑到安全和性能的问题。
